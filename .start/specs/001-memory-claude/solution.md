# SDD — Memory + CLAUDE.md System (M2)

## Architecture Overview

The system has three layers that work together:

```
Layer 1: Structure
  docs/ai/memory/          ← visible memory bank in repo
  CLAUDE.md (root + dirs)  ← lean, routing rules, @imports

Layer 2: Skills Pipeline
  memory-route             ← capture: extend /reflect routing
  memory-sync              ← maintenance: keep @imports in sync
  memory-cleanup           ← maintenance: prune, archive
  memory-promote           ← growth: domain → tcs-patterns skills

Layer 3: Onboarding
  tcs-helper:setup         ← generates Layer 1 for new repos
```

No MCP dependency. All state in markdown files. Skills are invoked manually or by hook.

## 1. Memory Structure

### 1.1 Repository Layout

```
repo-root/
├── CLAUDE.md                     # lean root (< 100 lines)
│                                 # contains routing rules + @imports
├── docs/
│   ├── CLAUDE.md                 # docs-area rules (when working in docs/)
│   ├── ai/
│   │   ├── CLAUDE.md             # memory bank maintenance rules
│   │   └── memory/
│   │       ├── memory.md         # INDEX — ≤ 200 lines, loaded at session start
│   │       ├── general.md        # repo conventions, code style, naming
│   │       ├── tools.md          # CI, build tools, API clients, local scripts
│   │       ├── domain.md         # business rules, data models, arch decisions
│   │       ├── decisions.md      # ADR shorts or links to docs/adr/
│   │       ├── context.md        # current focus, sprint goals, active work
│   │       └── troubleshooting.md # known issues + proven fixes
├── src/
│   └── CLAUDE.md                 # code-area rules (TDD, patterns, imports)
└── test/
    └── CLAUDE.md                 # test-area rules (coverage, naming, structure)
```

### 1.2 Category Definitions

| File | Lifetime | What goes here |
|---|---|---|
| `general.md` | longlived | Naming conventions, folder structure, code style rules, git workflow |
| `tools.md` | longlived/medium | Build pipeline, CI commands, API client quirks, local dev setup |
| `domain.md` | medium | Business rules, data models, key entities, domain language |
| `decisions.md` | medium | Architecture choices + rationale (ADR format or pointer to `docs/adr/`) |
| `context.md` | short | Current sprint focus, active features, known blockers |
| `troubleshooting.md` | short | Known issues + fixes; archived when resolved |

### 1.3 Scope × Lifetime Mapping

| Scope | Lifetime | Location | Managed by |
|---|---|---|---|
| Global | static longlived | `~/.claude/includes/memory-preferences.md` | Auto-memory (existing) |
| Global | longlived | `~/.claude/includes/memory-*.md` | Auto-memory (existing) |
| Project | longlived/medium | `~/Kouzou/projects/<proj>/` | Auto-memory + manual |
| Repo | medium | `docs/ai/memory/{general,tools,domain,decisions}.md` | memory-route + memory-sync |
| Repo | short | `docs/ai/memory/{context,troubleshooting}.md` | memory-route + memory-cleanup |
| Session | really short | context window / auto-memory | M5 (MCP, future) |

## 2. CLAUDE.md Design

### 2.1 Root CLAUDE.md (< 100 lines)

Structure:
```markdown
# [Project Name]

## Core Philosophy
[2-3 lines: what this project is, key principles]

## Memory & Context
@docs/ai/memory/memory.md    ← always loaded (index + routing)

## Routing Rules
<!-- Where new learnings go — must be here, not in memory.md -->
- Personal/workflow corrections → auto-memory (global)
- Repo conventions/style → docs/ai/memory/general.md
- Tool/CI/build knowledge → docs/ai/memory/tools.md
- Domain/business rules → docs/ai/memory/domain.md
- Architectural decisions → docs/ai/memory/decisions.md
- Current focus/blockers → docs/ai/memory/context.md
- Bugs/fixes → docs/ai/memory/troubleshooting.md

## Active Skills
[List relevant tcs-* skills for this repo]

## Stack-Specific Rules
[Generated by tcs-helper:setup based on detected stack — TypeScript, Go, etc.]
```

### 2.2 Per-Directory CLAUDE.md

Loaded only when working in that directory (Claude Code recursive discovery).

**`src/CLAUDE.md`** — code area:
- TDD rules (RED-GREEN-REFACTOR, test-first)
- SDD contract reference (link to docs/ai/memory/domain.md)
- Import/module conventions
- Architecture patterns (links to tcs-patterns skills if installed)

**`test/CLAUDE.md`** — test area:
- Test file naming and folder conventions
- Coverage expectations
- Test data patterns
- Mutation testing guidance (if tcs-patterns:mutation-testing installed)

**`docs/CLAUDE.md`** — docs area:
- Documentation structure rules
- When to create new docs vs. update memory files
- ADR creation workflow (if tcs-team:record-decision available)

**`docs/ai/CLAUDE.md`** — memory bank rules:
- How to maintain memory files (when to update, how to keep lean)
- Category definitions (what goes in each file)
- When to run memory-cleanup vs. memory-promote
- Index (memory.md) maintenance rules

## 3. Skills Pipeline

### 3.1 `tcs-helper:memory-route`

**Invocation:** `/memory-route` (or `/tcs-helper:memory-route`)

**Purpose:** Extend `/reflect` with repo-layer category routing. Analogous to `/miyo-reflect` for MiYo repos.

**Trigger:** Run after `/reflect` in any TCS repo. Can also be invoked standalone with manual input.

**Input sources (in priority order):**
1. **From `/reflect` queue file** — reads `~/.claude/learnings-queue.json` (same file `/reflect` reads) filtered to repo-specific entries (entries whose scope is the current repo path). This is the primary handoff mechanism.
2. **From session output** — user can paste `/reflect` output text as argument: `/memory-route "use fd not find"`
3. **From `$ARGUMENTS`** — inline text: `/memory-route "always return null for unknown emails in UserRepository"`

**`learnings-queue.json` schema (claude-reflect contract):**
```
{
  "learnings": [
    {
      "text": string,           // learning content
      "scope": string | null,   // repo absolute path, or null for global
      "timestamp": string,      // ISO 8601
      "confidence": "high" | "medium"
    }
  ]
}
```
`memory-route` reads this file and filters for entries where `scope` matches the current working directory (exact prefix match). If the file does not exist or is empty, falls back to manual/argument input. **Implementation note:** Verify this schema against the actual claude-reflect source before building the queue reader (Phase 2 research task).

**Learning type:**
```
Learning {
  text: string          // the learning content
  source: string        // "reflect-queue" | "manual" | "argument"
  scope?: string        // repo path if known (from reflect queue)
  confidence: "high" | "medium"   // high = from queue, medium = manual
}
```

**Logic:**
```
For each learning:
  1. Check if already present in target file (deduplication):
     - Semantic similarity check (same fact, different wording = duplicate)
     - If duplicate found: skip silently, note in report
  2. Classify → category:
       conventions, naming, style        → general.md
       CI, build, deploy, scripts, tools → tools.md
       domain, entity, rule, model       → domain.md
       decision, chose, tradeoff, why    → decisions.md
       current, focus, working on        → context.md
       bug, error, fix, workaround       → troubleshooting.md
  3. Append to docs/ai/memory/{category}.md with date prefix
  4. Update memory.md index last-updated date for that file
```

**Override:** If a learning doesn't fit any category, invoke AskUserQuestion to select target file.

**YOLO mode:** When `$CLAUDE_YOLO_MODE` is set or `--yolo` argument is passed:
- Write learning to target category file as normal (**simultaneous write** — not staged-only)
- ALSO append the same entry to `docs/ai/memory/yolo-review.md` (audit log for the session)
- `yolo-review.md` is a human-reviewable record of all YOLO-session writes
- User runs `/memory-route --review-yolo` to inspect staged entries and selectively remove any from their category files if unwanted

**Interface:**
```
State {
  learnings: Learning[]       // from queue or manual input
  routed: RoutedLearning[]    // category + file + content
  skipped: Learning[]         // duplicates detected
  unclassified: Learning[]    // needs user decision
}
```

### 3.2 `tcs-helper:memory-sync`

**Purpose:** Keep `@imports` in root CLAUDE.md synchronized with actual memory files. Enforce 200-line budget.

**Trigger:** Run periodically, or when new memory files are added/renamed.

**Logic:**
1. Scan `docs/ai/memory/` for all .md files
2. Verify root CLAUDE.md has `@docs/ai/memory/memory.md` import
3. Verify `memory.md` index lists all category files (no orphans, no stale entries)
4. Check memory.md line count — warn if approaching 200
5. Verify routing rules are in CLAUDE.md (not duplicated in memory.md)

### 3.3 `tcs-helper:memory-cleanup`

**Purpose:** Reduce token load by pruning stale entries and archiving resolved issues.

**Trigger:** Run periodically (e.g. monthly, or when memory.md approaches 200 lines).

**Logic (per centminmod cleanup pattern):**
1. `troubleshooting.md`: entries marked "resolved" → move to `docs/ai/memory/archive/YYYY-MM/troubleshooting.md`
2. `context.md`: entries older than 2 weeks without update → flag for removal or archive
3. `domain.md` + `general.md`: identify duplicates, consolidate overlapping entries
4. `decisions.md`: entries that describe superseded decisions → mark as superseded, archive if very old
5. Update `memory.md` index after each operation

**Preservation rules (from centminmod):**
- Never delete todos or roadmap items
- Never delete historical decisions (only archive superseded ones)
- Only archive troubleshooting when explicitly marked resolved

### 3.4 `tcs-helper:memory-promote`

**Invocation:** `/memory-promote` (or `/tcs-helper:memory-promote`)

**Purpose:** Detect mature domain patterns and promote them to `tcs-patterns` skill candidates.

**Trigger:** Run periodically, or when `domain.md` grows large.

**Session history availability:**
- If `~/.claude/projects/<repo>/` contains session files → use reflect-skills analysis for cross-session evidence (confidence boosted)
- If session history is absent or sparse (< 3 sessions) → analyse `domain.md` alone using semantic pattern detection; all candidates default to confidence = Low until history accumulates

**reflect-skills analysis (internal):** Scan `~/.claude/projects/<repo>/` for session transcript files (`.jsonl`) to find patterns that repeat across sessions. This is a read-only file scan — no external tool or API call. If session files are absent, skip this step and proceed with domain.md-only analysis.

**Logic (uses reflect-skills analysis internally):**
1. Read `docs/ai/memory/domain.md`
2. Check session history availability (see above)
3. Scan available evidence for repeating domain-related patterns
4. Identify entries that:
   - Appear multiple times across sessions (if history available), OR
   - Describe a reusable architectural pattern (DDD, hexagonal, functional, etc.), OR
   - Are referenced frequently by other files
4. Propose as tcs-patterns skill candidate with evidence
5. On user approval:
   - If `tcs-patterns` plugin is installed: generate skill file stub in `plugins/tcs-patterns/skills/<skill-name>/SKILL.md`
   - If `tcs-patterns` is **not** installed: generate stub in `docs/ai/memory/promoted-skills/<skill-name>.md` with a note to move it once tcs-patterns is installed
   - Replace `domain.md` entry with pointer: `→ see tcs-patterns:[skill-name]` (or the promoted-skills path if tcs-patterns absent)
6. Update `memory.md` index

## 4. `tcs-helper:setup`

**Purpose:** One-shot project onboarding. Generates `docs/ai/` structure + CLAUDE.md files from stack detection.

**Trigger:** User invokes `/setup` in a new or existing repo.

**Logic:**
1. Detect tech stack (scan package.json, go.mod, Cargo.toml, pyproject.toml, etc.)
2. Detect existing patterns (DDD structure, hexagonal ports, CI provider)
3. Generate:
   - `docs/ai/memory/` directory + all 6 category files (with starter content)
   - `docs/ai/memory/memory.md` (index)
   - Root `CLAUDE.md` (from template, stack-customized)
   - `src/CLAUDE.md` (from template, stack-customized)
   - `test/CLAUDE.md` (from template, stack-customized)
   - `docs/CLAUDE.md`
   - `docs/ai/CLAUDE.md`
4. Apply stack-specific template overrides from `plugins/tcs-helper/templates/stacks/` based on detected stack (Cloudflare, Convex, TypeScript, Go, etc.)
5. Offer optional: create `docs/adr/` directory for architecture decision records
6. Offer optional: configure PostToolUse hooks (format on save for detected stack)

**Templates location:** `plugins/tcs-helper/templates/` (generic CLAUDE.md + category templates) and `plugins/tcs-helper/templates/stacks/` (stack-specific overrides, one file per stack + `generic.md` fallback).

## 5. Integration with claude-reflect

`memory-route` is the TCS repo-layer extension of `/reflect`. The full post-session flow:

```
/reflect          ← claude-reflect base (unchanged): processes queue, routes to
                    global CLAUDE.md / auto-memory / rules/*.md

/memory-route     ← TCS extension: takes repo-specific learnings from reflect
                    output and routes them to docs/ai/memory/ category files

/memory-promote   ← periodic: scan domain.md for promotable patterns
```

`/reflect` handles global and project-level routing. `/memory-route` handles repo-level routing. These do not overlap — `/reflect` already knows not to write repo-specific facts to global files; `/memory-route` picks up what `/reflect` left as "repo-specific".
